# v0.5.0 — User Configuration (TOML)

## Objective

Add a TOML-based user configuration system so users can customize CLI behaviour (paths, logging, API endpoints, policies) without passing flags every time.

## Design decisions

| Decision | Choice | Rationale |
|---|---|---|
| Config format | **TOML** | Human-readable, stdlib `tomllib` (Python 3.11+) — no new deps |
| Config path persistence | **Pointer file** at `~/.minecraftmodpack/config_location` | Fully automatic, works identically on Windows/Linux/macOS |
| Config directory | **`~/.minecraftmodpack/`** | Simple, works everywhere: `/home/user/`, `/Users/user/`, `C:\Users\user\` |
| `config init` command | **Yes** | Generate a commented default config the user can edit |
| Zero-config behaviour | **Built-in defaults match current hardcoded values** | `AppConfig()` with no args = exact same behaviour as today |
| Env var | **Not used** | Pure pointer file approach (per user preference) |

## Architecture

### Separation of concerns

```
                   ┌───────────────────────┐
                   │   FilesystemService   │  ← reads bytes from disk
                   │  .read_config(path)   │
                   │  .write_pointer(...)  │
                   └─────────┬─────────────┘
                             │ raw dict
                             ▼
                   ┌───────────────────────┐
                   │    ConfigManager      │  ← pure parser, no I/O
                   │  .parse(dict)         │
                   │  .validate(AppConfig) │
                   │  .generate_default_   │
                   │    toml()             │
                   └─────────┬─────────────┘
                             │ AppConfig (typed dataclass)
                             ▼
                   ┌───────────────────────┐
                   │      APP / cli()      │  ← orchestrator
                   │  resolves path        │
                   │  loads or defaults    │
                   │  passes AppConfig     │
                   │  to UI / Builder      │
                   └───────────────────────┘
```

Key rule: **`ConfigManager` never touches the filesystem.** All config I/O lives in `FilesystemService` alongside its existing file operations.

### Config path resolution (priority descending)

1. **`--config <path>`** CLI flag — session override, **also saves to pointer file**
2. **`~/.minecraftmodpack/config_location`** pointer file — read its content, that's the config path
3. **`~/.minecraftmodpack/config.toml`** — default location
4. **Built-in hardcoded defaults** — `AppConfig()` with all defaults, always works

### Zero-config guarantee

Every single `@dataclass` field carries a default matching the current hardcoded behaviour. `AppConfig()` with no arguments = identical behaviour to v0.4.0. The system never crashes due to a missing config file — it simply uses defaults.

### MODIFY: `src/__main__.py`

Config orchestration — resolve path, load or default, pass to APP:

```python
CONFIG_DIR = Path.home() / ".minecraftmodpack"
POINTER_FILE = CONFIG_DIR / "config_location"
DEFAULT_CONFIG = CONFIG_DIR / "config.toml"


def _resolve_config_path(cli_config: Path | None) -> Path:
    """Returns resolved config path using the 4-step chain."""
    if cli_config:
        FilesystemService.write_pointer(POINTER_FILE, cli_config)
        return cli_config
    pointer = FilesystemService.read_pointer(POINTER_FILE)
    if pointer:
        return Path(pointer)
    return DEFAULT_CONFIG


def _load_or_default(config_path: Path) -> AppConfig:
    if config_path.exists():
        raw = FilesystemService.read_config(config_path)
        return ConfigManager.parse(raw)
    return ConfigManager.get_default_config()


@click.option("--config", type=clickPath(dir_okay=False, path_type=Path),
              help="Path to TOML configuration file")
@version_option(version=version, message="%(version)s")
@group(cls=APP)
@click.pass_context
def cli(ctx, config, **kwargs):
    config_path = _resolve_config_path(config)
    app_config = _load_or_default(config_path)
    ctx.obj = {"config": app_config, "config_manager": ConfigManager()}
```

## Error handling

A `ConfigError` class **already exists** at `src/utils/errors.py` line 60 (extends `AppError` with `cause`, `context`, `code`). The Plan uses it directly — no new error type needed.

### Error propagation pattern

Errors follow the same three-layer pattern as the rest of the codebase:

```
Layer               Raises                      Caught by
─────               ──────                      ────────
ConfigManager       ConfigError                 cli() in __main__.py
FilesystemService   FilesystemError             cli() in __main__.py
APP / commands      (use _handle_app_error)     _handle_app_error() in APP
```

### Complete failure scenario table

Every config-related failure path, categorised by layer.

#### Config discovery & pointer file (in `cli()` orchestrator)

| Scenario | Error type | Error code | Context | What user sees |
|---|---|---|---|---|
| `~/.minecraftmodpack/` can't be created (perms) | `FilesystemError` | `config_dir_create_failed` | `{"path": "~/.minecraftmodpack"}` | Error message + fallback to defaults |
| Pointer file exists but unreadable (perms) | `FilesystemError` | `pointer_read_failed` | `{"path": "~/.minecraftmodpack/config_location"}` | Warning + fall through to default path |
| Pointer file content is not a valid path | `ConfigError` | `pointer_invalid_path` | `{"pointer": "<content>"}` | Warning + fall through to default path |
| `--config` path can't have pointer saved (perms) | `FilesystemError` | `pointer_write_failed` | `{"path": "~/.minecraftmodpack/config_location"}` | Warning (non-fatal, config still loaded for session) |
| `--config` flag points to non-existent file | `ConfigError` | `config_not_found` | `{"path": "/given/path"}` | Warning + defaults used + pointer still saved for future |

#### Config file reading (in `FilesystemService.read_config()`)

| Scenario | Error type | Error code | Context | What user sees |
|---|---|---|---|---|
| File not found | (not an error) | — | — | Silent — `_load_or_default()` returns `AppConfig()` |
| TOML syntax error | `ConfigError` | `invalid_toml` | `{"path": "...", "line": 12, "column": 5}` | Error with file path, line, column, and parser message |
| TOML syntax error causes unavailable in Python 3.13 | `ConfigError` | `invalid_toml` | `{"path": "..."}` | Error with file path and parser message |
| Binary file instead of text | `ConfigError` | `invalid_toml` | `{"path": "..."}` | Error, decode failure |
| Permission denied | `FilesystemError` | `config_read_permission` | `{"path": "..."}` | Error, hint to check file permissions |

#### Config parsing (in `ConfigManager.parse()`)

| Scenario | Error type | Error code | Context | What user sees |
|---|---|---|---|---|
| Section is not a table (e.g. `[logging] = "string"`) | `ConfigError` | `section_not_a_table` | `{"section": "logging", "type": "str"}` | Error with section name |
| Field has wrong type (e.g. `bar_width = "wide"`) | `ConfigError` | `field_type_mismatch` | `{"section": "ui", "field": "bar_width", "expected": "int", "received": "str"}` | Error with field path and expected type |
| Unknown field in a section | `ConfigError` | `unknown_field` | `{"section": "ui", "field": "colour"}` | Warning (forward compatibility — logged but **not** fatal) |
| Unknown section at root | `ConfigError` | `unknown_section` | `{"section": "my_custom_stuff"}` | Warning (logged but not fatal) |
| `logging.level` not in valid set | `ConfigError` | `invalid_enum_value` | `{"section": "logging", "field": "level", "valid": ["debug","info","warning","error"], "received": "verbose"}` | Error with valid options listed |
| `logging.target` not in valid set | `ConfigError` | `invalid_enum_value` | `{"section": "logging", "field": "target", "valid": ["console","file","both"], "received": "socket"}` | Error with valid options listed |
| `paths.logs_dir` is not a string | `ConfigError` | `field_type_mismatch` | `{"section": "paths", "field": "logs_dir", "expected": "string", "received": "int"}` | Error |
| Section missing entirely | (not an error) | — | — | Silent — `config_from_dict` defaults the section |
| Field missing from a section | (not an error) | — | — | Silent — field gets its dataclass default |

#### Config writing / `config init` (in `FilesystemService.write_config()` and `cli/app.py`)

| Scenario | Error type | Error code | Context | What user sees |
|---|---|---|---|---|
| `config init` target exists, no `--force` | (not an error) | — | — | Warning with path + "use --force to overwrite" |
| Can't write generated config (perms) | `FilesystemError` | `config_write_failed` | `{"path": "..."}` | Error |
| Can't write generated config (disk full) | `FilesystemError` | `config_write_failed` | `{"path": "..."}` | Error |
| Parent directory doesn't exist for `config init` | `FilesystemError` | `config_dir_create_failed` | `{"path": "..."}` | Error |
| `config init` —force and write succeeds | (success) | — | — | Success message with path |

#### Runtime use of config values

| Scenario | Error type | Error code | Context | What user sees |
|---|---|---|---|---|
| `paths.logs_dir` can't be created (perms) | `FilesystemError` | `log_dir_create_failed` | `{"path": "..."}` | Error during APP init, fallback to `cwd/logs` |
| `modrinth.base_url` is unreachable | `ModrinthError` | `request_failed` | (existing handling) | Existing error path |
| `filesystem.hash_chunk_size` is 0 or negative | `ConfigError` | `invalid_value` | `{"section": "filesystem", "field": "hash_chunk_size", "reason": "must be > 0", "received": "0"}` | Warning + clamped to 8192 |

### Summary of `ConfigError` usage

```
errors.py (already exists):
  ConfigError(AppError):    # code, context, cause
    codes used:
      - invalid_toml          TOML syntax error
      - invalid_value         Value semantically wrong (e.g. hash_chunk_size ≤ 0)
      - invalid_enum_value    String not in allowed set
      - field_type_mismatch   Python type mismatch
      - section_not_a_table   Section is a scalar instead of a table
      - unknown_field         Unknown key in a section (warning)
      - unknown_section       Unknown top-level section (warning)
      - pointer_invalid_path  Pointer file content is garbage
      - config_not_found      --config points to missing file
```

### How errors surface to the user

Errors during config discovery/loading happen **before** the command callback runs — they happen in `cli()` (the group callback) or in `APP.__init__()`. There is no active stage/logger at that point, so errors are printed directly via `click.echo()` or `sys.stderr`:

```python
# In cli() group callback (__main__.py):
try:
    config_path = _resolve_config_path(cli_config)
    app_config = _load_or_default(config_path)
except (ConfigError, FilesystemError) as exc:
    click.echo(f"Config error: {exc}", err=True)
    app_config = ConfigManager.get_default_config()  # graceful fallback
```

Once inside a command callback, config errors that surface at runtime (e.g. `logs_dir` can't be created) go through the existing `self._handle_app_error()` path, which prints the error via `UI.error()` and logs the full context.
